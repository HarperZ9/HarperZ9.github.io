EMET launch

An external byte-witness for agentic work.

EMET re-derives sha256 over source bytes and reports one of three deliberately limited verdicts: MATCH, DRIFT, or UNVERIFIABLE. Its only credential is re-derivation, never authority. There is no TRUSTED verdict, by construction.

Byte-witness

What an external byte-witness is, and why agentic work needs one.

01
External, not in-band

It re-derives the hash itself.

A model or agent acts on bytes: a file it edits, a prompt it is handed, a generated view it presents as faithful to source. EMET reads the raw bytes through open('rb'), never a mediated view, and re-computes the sha256 over those source bytes. The verdict comes from re-running the same code on the same bytes, not from a system vouching for itself. EMET runs outside whatever it audits.

02
The gap it covers

Build provenance does not catch drift.

Build-provenance tooling does not cover a system vouching for itself in-band, a presented view drifting away from source, or a monitor reading a different file than the one that actually runs. Those are exactly the failures that grow as agents act faster than ordinary review can absorb. EMET is the small external check that the presented material still matches the source it claims to represent.

self-vouchingview driftread-path divergencehandoff

Why it matters: EMET sits on the accountability seam. WARDEN carries authorized scope and constraints inward; EMET carries a verifiable, re-derivable work record outward. The witness does not decide whether a model is safe and grants no permission. It reports what was compared and what verdict was produced, so a reviewer, not the witness, owns the trust decision. emet is Hebrew for "truth."

Verdict lattice

A closed lattice with no authority verdict.

MATCH

Comparison holds

The checked material re-derived to the expected byte-level result. Absence of DRIFT is MATCH or UNVERIFIABLE, never trust.

DRIFT

Bytes or view changed

The presented view or the checked bytes no longer re-derive to what was anchored. Drift is reported as a fact, not adjudicated as a fault.

UNVERIFIABLE

Witness cannot compare

There is no raw-byte channel, no anchor, or no second read path. EMET reports inability with a stable reason code; it never substitutes a default or crashes.

Closed by construction: the primary integrity lattice is exactly {MATCH, DRIFT, UNVERIFIABLE}. Auxiliary channels (coherence, corroboration, the audit chain, the monitor and organs judgements) are each their own closed set, and none of them is or maps to TRUSTED. The spec declares the lattice closed; the code makes that closure structural rather than a matter of review.

Structural refusal

governed() refuses authority tokens at construction.

01
Construction-time guard

An authority verdict is a hard error.

Every governed verdict is emitted through governed(channel, token). It asserts the token belongs to its channel's closed set and returns it verbatim, so the emitted bytes are unchanged. If a codepath tries to emit TRUSTED, APPROVED, SAFE, ALLOWED, AUTHORIZED, or any other forbidden authority word, it raises VerdictError at construction time, inside the trusted core, before a byte reaches stdout. An unsanctioned verdict is a construction error, not a review miss.

02
In-band claims refused

It strips in-band authority claims.

EMET also refuses authority claims embedded in the material it checks. The refuse command scans bytes against a versioned marker corpus, writes a clean copy with the in-band authority claims neutralized, and logs how many were found. The denylist is a set of known signatures, not a proof of completeness, and EMET says so: absence of a flag is not a guarantee of cleanliness.

marker corpusdetect and stripobeyed: nonehonest denylist

Accountability, not its opposite: refusing to emit authority and refusing to obey in-band authority claims are the structural inverse of a tool that hides intent or manufactures permission. EMET makes the comparison transparent and auditable; it never grants authorization and never bypasses a safeguard.

Tamper-evident log

Attestations chain by hash. Retroactive edits show.

Each recorded fact (anchor, verify, coherence, refuse, corroborate) is appended to a log where every entry carries the hash of the entry before it: chain = sha256(prev + kind + fact). The audit command recomputes the whole chain from a fixed genesis value and reports a single closed result: INTACT or BROKEN. A retroactive edit anywhere in the log breaks the chain at that point, and the audit names the first broken entry.

What chains

kind, fact, and the previous entry's chain hash, recomputed deterministically from source bytes.

What audit reports

A closed two-value result, INTACT or BROKEN, plus the entry index where the chain first breaks. No authority word appears in the result.

Three-language conformance

Re-derived across three implementations, with the caveat stated plainly.

Python

Reference

A stdlib-only reference (membrane.py / organs.py / monitor.py / corpus.py), no dependencies.

Rust

Second impl

A from-scratch Rust implementation, no crates, built from the spec text.

Node.js

Third impl

A clean-room Node.js implementation using built-in modules only, no npm.

Honest caveat: all three implementations pass the same 19 conformance vectors in CI on every push (byte-hash core, marker path, and the audit chain, write and verify). But all three share an author. That agreement shows the spec is implementable from its text alone, and the result is re-derivable in principle by a different author following the spec. It does not yet show independent different-author re-derivability. The spec names a different-author implementation as the open proof gate, and EMET makes that gap prominent rather than papering over it. For a tool whose only credential is reproduction, an inflated claim would refute itself.

Proof surface

The proof surface builds outward from EMET.

01
Witness receipt

EMET emits a compact witness receipt.

An optional adapter outside the EMET core emits compact JSON witness receipts for proof-index and release-readiness workflows. It does not change governed stdout, signing, enforcement, or actuation boundaries. The adapter only accepts whole governed verdict tokens and refuses authority-shaped stdout before it enters a receipt, mirroring the same closed-lattice discipline as the core.

02
Shared contract family

proof-surface mirrors the witness.

proof-surface is a stdlib-only contract family: a proof-surface packet, a hard-pinned work-record receipt that is the structural inverse of an authorization-suppression prefire, and a witness-receipt validator that mirrors EMET's discipline. It is covered by 21 tests. From there, proof-surface-report adapts packets and receipts to reviewer Markdown and rejects authority-shaped language, and repo-proof-index aggregates packets and receipts into a reviewer-ready index.

proof-surface packetwork-record receipt21 testsreviewer Markdown

Bilateral provenance: trusted context is carried inward toward the model; a verifiable, re-derivable work record is carried outward toward the reviewer. EMET is the witness anchor of that outward record. The work-record receipt is a development work record, not a shipped customer-facing trust artifact, and the proof surface presents it as such.

Position

A witness is not an authority.

EMET is the accountability spine for AI-assisted work: re-derive the hash over source bytes, emit a closed MATCH/DRIFT/UNVERIFIABLE verdict, refuse every authority token at construction, and chain each fact into a tamper-evident log. It is re-derivable in principle from the spec alone, demonstrated today across three same-author implementations; an independent different-author implementation is the named next step. The public repo shows the method in full; it does not ask the reader to accept anything on faith.
