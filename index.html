PROJECT TELOS · LET’S HAVE A LOOK

AI that shows its work — so you can trust it with yours.

Every AI tells you things, and sounds just as sure when it’s wrong. These are small, plain tools that fix that: the AI looks at the real thing, shows you exactly what it saw, and you can re-run the check yourself and get the exact same answer every time. So instead of hoping it got it right, you and the AI can look at the same thing together — and verify every step. The checking is the floor — what it lets you and the AI build, together, is the point.

You don’t have to take my word for this — I don’t take my own. Here is the machinery.

Four things, taken straight from the running tools: one that rebuilds a file from scratch to prove not a single byte was changed; one that refuses to use a part it can’t fully account for; the plain card every tool fills out about itself; and those pieces joined into one loop — a model looks at a live page and, in the same breath, is stopped from doing something it wasn’t allowed to. Each one shows its real history, and its real limits.

Underneath all four is one plain habit: I don’t ask you to trust the thing in front of you — I give you a way to check it. You take what something shows you and hold it against a standard it didn’t get to set for itself, then keep a proof you can run again yourself and get the exact same answer — the way a music box plays the very same tune, note for note, each time you turn it, because the tune is read off the pins and never performed by someone who might waver. The number it starts from is those pins; anyone who runs it gets the identical result. When there’s nothing solid to check it against, it says so — UNVERIFIABLE — instead of pretending to be sure. And catching a forgery turns out to be the same move as making something genuinely new: one checks a thing against where it came from, the other against everything made before it.

Exhibit I EMET · the witness re-derives the bytes
$ python membrane.py verify demo_sample.txt
MATCH  demo_sample.txt   want=d02206abca2a…  got=d02206abca2a…   (exit 0)

# append one byte to the file, ask the same question
$ python membrane.py verify demo_sample.txt
DRIFT  demo_sample.txt   want=d02206abca2a…  got=b072d611645e…   (exit 2)

# one spec, three separate language implementations, CI on every push
membrane.py        →  CONFORMANCE 19/19 vectors pass
impl/rust/emet     →  CONFORMANCE 19/19 vectors pass
impl/js/emet.js    →  CONFORMANCE 19/19 vectors pass

Change one byte of the source and the verdict falls from MATCH to DRIFT. The seal answers to the bytes, not to whoever vouches for them.

Captured live from the emet run · receipt format from adapters/proof_surface_receipt.py · 19/19 conformance vectors, three language implementations, CI each push. Honest limit: all three share one author — this shows the spec is implementable from its text, not yet independently re-derived. A fourth, different author’s implementation is owed.

Exhibit II QuantaLang · the compiler refuses what it can’t account for
fn pick<'a, 'b>(x: &'a i32, y: &'b i32) -> &'a i32 {
    y          // returns 'b where the signature promised 'a
}

$ quantac check pick.quanta
Lexing... OK (28 tokens)
Parsing... OK (1 items)
Type errors found:
  function returns `&'b i32` but expected `&'a i32`

The function promises the memory tied to ‘a; it tries to hand back ‘b’s instead, and the compiler refuses. It tracks whose state crosses each boundary and won’t let one escape the contract it was given.

Captured live from quantac check (Rust toolchain, release build) · diagnostic at compiler/src/types/unify.rs · 999 tests green · C is the shipped end-to-end backend. QuantaLang is foremost an effects language: what a function may do is typed like whose memory it returns — the same accountability, two faces.

  [perception] web-document perceived  ·  1 link, 52 text chars
  [decision] summarize → allow     [decision] delete → deny

A model perceives a live page natively — witnessed, digest-stamped, not a screenshot — and the unauthorized act is refused by the gate, not by a prompt. Awareness and restraint as one mechanism.

Now a real, growing tool — github.com/HarperZ9/accountable-surface, 129 tests, an MCP server, and filesystem / web / command effectors that stay inert until the gate allows. The run above is the decision layer; in the current build an allow is carried out by a bounded, reversible effector and then re-perceived to confirm it. Honest limit: still not folded into ORCA, and the gate stays advisory until a runtime enforces it. The loop runs; the integration is owed.

More than a receipt — a place to look, together.

The checking isn’t the end of it. Because the tool actually perceives the real thing — a file, a running system, an image, live data — and shows you exactly what it saw, you and an AI can work over the same live state at the same time. You point at it, ask why it thinks what it thinks, make it re-check. It can’t bluff, because you’re both looking at the same verified view.

“The AI told me” becomes “the AI and I looked at it together — and I can verify every step.”

That’s what makes it safe to use for real work, in any field: a designer and an AI reading the same render; a researcher and an AI examining the same data; an engineer and an AI watching the same live system — places where nothing can be taken on faith. Not trust, but checking, live, together. And because every look is witnessed and every action is gated and re-checked, you can let it act, not just talk.

The reasoning underneath

A model infers the world. It never senses it.

Every AI model reaches you the same way: it just tells you things. Not from a memory it actually keeps, or events it actually watched happen — and it sounds exactly as sure either way. For years we answered a system that can’t check itself by making it bigger and smoother, which mostly made it better at sounding sure. Even the trade we’ve all accepted as a law of nature — that an AI can be capable or safe, but never both — turns out to come from this one blind spot. A mind you can’t see into, you have to keep on a short leash. A mind that can’t see, you can’t really trust.

The gap is not intelligence. It is a sense organ.

So I built a sense organ you can run.

A creative engine with real senses. It sees — a live camera, read as brightness, edges, and motion, actually measured rather than guessed, and it shows you what it’s seeing right beside the art, so you can check. It creates from what it sees — the spiral a sunflower packs its seeds in, flowing fields, branching veins, the patterns two spreading chemicals make, slime-mould networks, fractal curves, lightning, crystal tilings, and a couple dozen more, all drawn live along the edges in front of the lens. And it keeps itself honest — every drawing grows from a single starting number, so the same number always redraws the exact same thing; saving the file has to pass the gate; and anyone can re-run it and get an identical result. That’s stronger than a signature, because it never asks you to trust the signer. Senses first, then a way to check, then freedom — made into something you can hold in your hand.

Accountability is the floor, not the ceiling. Once the proof actually holds, the limits are free to move.

The surface.

“Surface” is just my plain word for one idea. Every tool here is a different creature — one writes programs, one draws pictures, one mixes colors, one keeps proofs, one guards a boundary — but each answers the same six questions about itself, the way a passport answers the same few questions at every border, no matter which country issued it. Not a platform they’re built on top of; a shared shape they all take, so very different tools fit together without blurring into one.

CARRIER what holds the state · text, image, shader frame, packet, model output, repo
BOUNDARY where it crosses · model, API, process, GPU, operator, scope-of-law
CHANNEL how it moves · CLI, HTTP, MCP, filesystem, git, the covert path
TRANSFORM what changes it · compile, render, redact, infer, verify
WITNESS how the change is proven · hash, receipt, provenance, frame trace, test
SCOPE who allowed it, in what mode · dry-run, fixture, local lab, authorized, operator-live
Answer those six, and a tool can connect to everything else without losing what makes it itself. The tools stay different; the shape they share is what holds them together.

Why it’s built this way.

The same discipline that makes the tool check itself is what keeps it from being misused. By design, its authority can only serve the thing it’s responsible for — it can’t be turned against it. And that lives in the code, where you can check it, not in a promise from me; my word is worth no more than anyone else’s.

It doesn’t stop at the machine. The same question turns on you: could you show your work, if someone asked? A tool that can account for itself, used by people willing to do the same. I’d rather this be ours than an authority over anyone — accountability you choose, not one handed down.
