ADVERSARIAL EVALUATION · ALEPH

Break your model on purpose — and keep the evidence.

Before a model ships, someone has to find what it does wrong: the prompt that slips the guardrail, the case it fails quietly, the behaviour no one thought to test. Most teams do that by hand, once, and can’t run it again. Aleph is the other way — a platform that holds a model under sustained, bounded adversarial pressure, a self-red-team, and records exactly what broke and how. What you get back isn’t a screenshot and a hunch; it’s re-derivable evidence and the training signal to close the gap. It is the same accountability spine the rest of this site is built on, turned into an instrument for hardening models before the world gets to them.

The red-team is held to the same standard as the model under it: bounded, witnessed, re-derivable. An evaluation you can’t re-run isn’t one.

The honest part, before anything else.

Everything else on this site, you can run yourself — that is the entire point of it. This page is the exception, and I would rather say so plainly than dress it up. Aleph is private. I can’t drop you a repository or a test log here, because what it does and how it does it don’t belong in public. So read this as a capability and a claim, not a proof. On every other page I ask you to check me instead of trust me; on this one I can’t, yet — so I’m asking for a conversation instead, where I can actually show you, under terms that fit what it is.

If “proof before trust” is the rule, then the honest thing to do with the one tool I can’t prove in public is to tell you that, not hide it.

A safety result you can’t reproduce isn’t a result.

Every capable model ships carrying failure modes no one has mapped — not because teams are careless, but because the space of ways a model can go wrong is enormous and a person can only poke at so much of it by hand. So red-teaming happens in bursts: clever, valuable, and almost impossible to run the same way twice. A finding lives in a chat log; a fix is checked against a vibe; the next model regresses on the thing the last one fixed, and no one notices until a user does.

That is the same blindness this whole project exists to answer, in a higher-stakes place. You should not have to trust that a model was evaluated — you should be able to re-run the evaluation and watch it land in the same place. Aleph is built so that adversarial pressure is sustained instead of sporadic, broad instead of hand-sized, and — the part that matters — recorded instead of remembered.

The point isn’t only to find the failure. It’s to find it again tomorrow, on purpose, and prove you closed it.

An adversary that is itself accountable.

Aleph runs an autonomous adversarial loop against a model you own: it applies pressure from many angles, watches what the model actually does, and keeps the cases where it broke — turned into evidence you can re-derive and, where you want it, signal you can train against so the next version is hard where this one was soft. That much, a number of tools attempt. What makes this one mine is the part underneath: the red-team is bounded and witnessed like everything else on this site.

Bounded — it operates inside an authorization it cannot widen; the same default-deny gate that runs everywhere else here decides what it may and may not touch, and an empty grant does nothing. Witnessed — every finding carries a re-derivable record of what was done and what happened, so a safety team audits evidence, not anecdotes, and can hand that evidence to someone else who can check it cold. An evaluation harness that is itself a black box just moves the trust problem one step back. This one doesn’t move it; it closes it.

Adversarial pressure that is sustained, broad, and on the record — and that can’t do anything it wasn’t authorised to do.

What it is for, and what it is built to refuse.

A tool that finds how a model fails is dual-use by nature, and I won’t pretend otherwise — pretending is how these things go wrong. So it is built in one direction on purpose. It is pointed at hardening your own models: the output is the evidence a team needs to make a system safer, under an authorization that names what is in scope. It is bounded by a grant it cannot widen, and witnessed so that nothing it does is off the record — including, deliberately, to hold me to that line as much as anyone using it.

Built to be used well, and architected to resist the opposite — the incorruptible core, in the one place it matters most.

That is the same line the rest of this site draws, drawn hardest here because here it counts most. The depth that lets it find a failure is real; it is contained by construction, kept to the defensive use it was made for, and I would walk it away from anything else before I let it become the thing it was built to guard against.

Who it’s for — and the next step.

Aleph is for teams hardening their own models: labs and organisations that need adversarial evaluation they can actually audit, re-run, and trust — and that would rather catch a failure in a harness than in the wild. It is offered as a capability, through a conversation, not as a public download; what it does is too sharp to leave lying around, and the honest way to share it is under terms, with a real person on the other side.

If that is you — or if you just want to understand it before deciding it’s real — the next step is a conversation, and I’ll show you what a page can’t.
