What a function may do belongs in its type.

QuantaLang is a typed-effects language. A function’s signature names the effects it may perform — reach the filesystem, the console, the network — and the lifetimes of the references it returns. The compiler checks both, and lowers the program to native code through a C backend. Perform an effect you never declared, and it does not compile.

I built it to make the compiler answer a question I couldn’t answer by hand — what is this function actually allowed to do? It’s tested where it counts, honest about its edges, and the discipline inside it runs underneath the rest of the work here.

quantac 1.0.0 · a Rust-built compiler · effects and lifetimes, in the type

The signature stops being a suggestion.

Most languages let a function reach out and touch the world — open a file, a socket — with nothing in its signature to say so. QuantaLang puts it in the type. A function carries the set of effects it may perform; call something that performs one it never declared, and the program is refused — with the exact call that gave it away named in the error. Memory works the same way: which reference comes back, and how long it lives, is checked, not assumed.

Two questions a type ought to answer — what may this do, and whose memory does it return — answered by the compiler, not the comment.

Don’t take the type. Watch it refuse.

Each of these is a real file checked by the real binary — quantac 1.0.0, release build. The output is verbatim.

Exhibit I QuantaLang · effects are part of the signature
fn load_config() {
    read_file("ops.toml");        // reaches the filesystem
}

$ quantac check missing_effect.quanta
function `load_config` performs effect `FileSystem` but does not declare it
  help: add ~ FileSystem to the signature
  note: capability `FileSystem` was triggered by ambient call(s): read_file

# declare the effect, and the same check passes — and writes a receipt
fn load_config() ~ FileSystem { read_file("ops.toml"); }

$ quantac check load_config.quanta --receipt receipt.json
Type checking... OK
  "declared_effects":      { "load_config": ["FileSystem"] }
  "observed_capabilities": { "load_config": { "FileSystem": ["read_file"] } }

Reach the filesystem without saying so and it won’t compile — the error names the call that gave it away. Declare the effect and it passes, and the compiler writes down what you claimed beside what it observed.

Captured live from quantac check (Rust toolchain, release build) · the effect is part of the function’s type, checked at every call. The receipt is an evidence contract — what was declared vs. what was observed — not a claim that any backend is production.

Exhibit II QuantaLang · memory has an owner, and the type tracks it
fn pick<'a, 'b>(x: &'a i32, y: &'b i32) -> &'a i32 {
    y          // returns 'b where the signature promised 'a
}

$ quantac check pick.quanta
Lexing... OK (28 tokens)
Parsing... OK (1 items)
Type errors found:
  function returns `&'b i32` but expected `&'a i32`

The signature promises the reference tied to ‘a; the body hands back ‘b’s. Refused. The lifetime is part of the contract, and the contract is enforced.

Diagnostic from compiler/src/types/error.rs, surfaced through the lifetime unification in unify.rs. Honest limit: borrow and lifetime checking is early — interprocedural, Phase 1; full caller-side borrow proofs are not yet claimed complete.

Exhibit III QuantaLang · checked, then compiled to native
# an algebraic effect with a handler — parsed, effect-checked,
# lowered to C, compiled, and run:
$ quantac run examples/quickstart/effects_greeting.quanta
Hello, teammate!

The same effect discipline rides all the way down — parse, type-check the effects, lower to C, run. One backend takes a program to a real native executable.

C is the shipped end-to-end backend; the test suite verifies it. The other targets — LLVM, WASM, SPIR-V, x86-64, ARM64, a Rust source lane — emit code but are experimental. Said plainly, not implied.

999 tests, and a short list of what doesn’t ship.

Built in Rust, with 999 passing tests run on every change and none failing — grown from the 868 the docs recorded on 2026-06-15. One backend, C, takes a program all the way to a native executable, and it is the path the suite proves end to end. That is the whole of what I claim — no more than the tests will stand behind.

It isn’t winding down. It’s holding still — because the rest is built on it.

The core is shipped and stable: effects and lifetimes in the type, checked to native C, are a foundation the rest of the work here stands on — not a product still chasing features. Several pieces are honestly partial, and I’d rather you hear that from me than find it: the self-hosted compiler is architecture, not yet a working build; async, garbage collection, and FFI are designed but not yet linked into compiled output; the experimental backends emit code but aren’t production. I built this alone. Those are the edges I know are there — the named limits of a foundation, not the closing of a project.

Build it. Then make it reject something.

Rust 1.70 or newer and a C compiler on the path. Clone, build, and quantac doctor confirms the C backend is ready. Point it at the examples — or write a function that lies about its effects, and watch it refuse.

$ git clone https://github.com/HarperZ9/quantalang && cd quantalang/compiler
$ cargo build --release
$ quantac doctor                       # toolchain + C-backend readiness
C backend: ready

$ quantac run examples/quickstart/effects_greeting.quanta
Hello, teammate!
